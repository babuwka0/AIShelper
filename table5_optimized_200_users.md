# Таблица оборудования и ПО для сети «Зубич» - оптимизированный вариант для компании до 200 сотрудников

Ниже приведен **второй, более сбалансированный вариант** проекта. Он сделан специально для учебного сценария, где:

- организация насчитывает около **200 сотрудников**;
- компания может **расти**, поэтому запас по производительности нужен;
- проект должен оставаться **официально реализуемым в России**;
- требования по отказоустойчивости, DR и резервному копированию сохраняются;
- при этом не хочется уходить в избыточный enterprise-overkill.

Главная идея этого варианта:

1. **Сохраняем правильную архитектуру**: Primary Site + DR Site + offsite/offline backup.
2. **Уменьшаем мощность серверного и storage-контура** до уровня, разумного для компании до 200 сотрудников.
3. **Не строим тяжелый ERP-контур**, если преподаватель считает его избыточным для учебного проекта.
4. **Оставляем запас на рост**, но без заведомо чрезмерной конфигурации.

---

## 1. Почему этот вариант более целесообразен

В первой версии был сделан акцент на максимально отказоустойчивой архитектуре при неограниченном бюджете. Это технически корректно, но для компании примерно на 200 сотрудников часть серверного оборудования и СХД получалась избыточной.

В этой версии:

- сокращено количество мощных серверных узлов;
- уменьшен объем RAM и storage-пула;
- DR Site ориентирован на **критичные сервисы**, а не на полное зеркалирование всей площадки в enterprise-масштабе;
- вместо тяжелого ERP-подхода делается упор на **специализированную стоматологическую МИС** и отдельные профильные сервисы.

## 3. Прикладной контур: что взять вместо «тяжелого ERP»

Для учебного проекта на 200 сотрудников **полноценный ERP-контур 1С действительно может выглядеть избыточно**, особенно если задача не про финансово-производственный холдинг, а про сеть стоматологий.

Поэтому здесь разумнее принять:

### Основные прикладные системы

| Функция | Рекомендуемое решение | Почему это лучше для учебного проекта |
|---|---|---|
| Основная медицинская система | **специализированная стоматологическая МИС** (например, 32top / Инфодент / аналог из реестра РФ) | это ближе к предметной области, чем ERP |
| Учет доступов | **ALD Pro** | отдельный системный сервис |
| БД | **Postgres Pro** | надежная российская СУБД |
| Почта | **МойОфис Почта** | не перегружает проект лишним функционалом |
| Документы | **Р7-Офис** | достаточно для повседневной работы |
| Service Desk | **Naumen Service Desk** | отдельный контур ИТ-поддержки |

### Как это можно сформулировать в работе

> Для учебного проекта вместо тяжелой ERP-системы целесообразно использовать специализированную стоматологическую МИС, так как она лучше соответствует бизнес-процессам клиники и не создает избыточной сложности архитектуры.

## 6. Backup и защита данных

Даже в оптимизированной версии это нельзя выбрасывать.

| Контур | Решение | Назначение |
|---|---|---|
| Локальный backup | **RuBackup** | быстрые локальные копии |
| Реплика backup на DR | **RuBackup replication** | защита от потери основной площадки |
| Offsite-копия | **TATLIN.OBJECT / S3 в РФ** | immutable/offsite защита |
| Offline-копия | **ленты / съемные носители / сейф** | защита от катастрофы и шифровальщиков |

### Политика backup

- схема `3-2-1-1-0`;
- инкрементальные копии ежедневно;
- полные копии еженедельно;
- offsite-копия по расписанию;
- offline-экспорт критичных архивов не реже раза в месяц;
- обязательная проверка восстановления.

## 10. Расчет энергопотребления и охлаждения серверной

Для учебного проекта полезно показать не только список оборудования, но и расчет инженерной нагрузки. В серверной почти вся потребляемая электроэнергия превращается в тепло, поэтому **1 Вт электрической мощности можно считать примерно равным 1 Вт тепловыделения**.

### 10.1. Расчетная нагрузка по основному серверному контуру

| Оборудование | Количество | Расчет на 1 шт. | Суммарная расчетная мощность |
|---|---:|---:|---:|
| **UserGate D200** | 2 | до **220 Вт** | **440 Вт** |
| **Eltex MES3324F** | 1 | **120-150 Вт** | **150 Вт** |
| **Aquarius AQserv T50 D212RS**, узел виртуализации | 3 | **450-550 Вт** рабочая / до **700 Вт** расчетная | **1650 Вт** |
| **Aquarius AQserv T50 D212RS**, узел БД | 2 | **400-500 Вт** рабочая / до **650 Вт** расчетная | **1000 Вт** |
| **Aquarius AQserv T50 D212RS**, backup-сервер | 1 | **300-400 Вт** рабочая / до **500 Вт** расчетная | **400 Вт** |
| **YADRO TATLIN.FLEX.TWIN** | 1 | **600-900 Вт** | **800 Вт** |
| Compact backup storage / дисковая полка | 1 | **300-500 Вт** | **400 Вт** |
| Мониторинг, KVM, датчики среды, служебная периферия | 1 комплект | **150-250 Вт** | **200 Вт** |

### 10.2. Итоги по мощности

- **Рабочая ИТ-нагрузка серверной:** около **5.0 кВт**
- **Пиковая / проектная нагрузка ИТ-контура:** около **6.0-6.5 кВт**
- **Рекомендуемый запас на рост:** **20-30%**
- **Проектная нагрузка с запасом:** около **8 кВт**

Такой запас нужен на:

- рост числа сотрудников и сервисов;
- добавление новых ВМ;
- расширение storage-пула;
- увеличение количества медицинских данных и файловых архивов.

### 10.3. Требования к охлаждению

Так как почти вся электрическая мощность превращается в тепло, для проектирования охлаждения принимаем:

- **ИТ-нагрузка с запасом:** около **8 кВт**
- **Минимальная требуемая холодопроизводительность:** **не менее 8 кВт**
- **Рекомендуемая проектная холодопроизводительность:** **9-10 кВт**

Для оптимизированного варианта разумно принять:

| Узел | Рекомендуемое решение | Пояснение |
|---|---|---|
| Основное охлаждение серверной | **2 прецизионных кондиционера по 8 кВт** | схема **N+1**, один работает, второй резервирует или участвует в ротации |
| Расчетная холодопроизводительность | **16 кВт суммарно установленной мощности** | этого достаточно, чтобы при отказе одного кондиционера второй держал серверную в допустимом режиме |
| Целевой режим | **22 +/- 2 °C**, влажность **40-60%** | безопасный режим для серверного оборудования |
| Контроль среды | датчики температуры, влажности и протечки | обязательно завести в мониторинг |

### 10.4. Требования к ИБП

Для проектной ИТ-нагрузки порядка **8 кВт** рекомендуется:

| Узел | Рекомендуемое решение | Пояснение |
|---|---|---|
| ИБП линия A | **online UPS 10 кВА / 9 кВт** | питание первой шины A |
| ИБП линия B | **online UPS 10 кВА / 9 кВт** | питание второй шины B |
| Время автономии | **10-15 минут** | достаточно для переключения и запуска ДГУ |
| Схема питания | **A/B PDU + два БП у серверов** | разделение нагрузки и отказоустойчивость |

### 10.5. Требования к дизель-генератору

При подборе ДГУ нужно учитывать не только серверы, но и:

- охлаждение;
- автоматику;
- освещение серверной;
- СКУД;
- мониторинг и вспомогательные инженерные системы.

Для оптимизированного варианта разумно принять:

| Узел | Рекомендуемое решение | Пояснение |
|---|---|---|
| ДГУ | **дизель-генератор 25-30 кВт** | покрывает ИТ-нагрузку, инженерные системы и один рабочий кондиционер |
| Режим работы | аварийное питание серверной и инженерки | для длительной автономной работы |
| Связка с ИБП | **АВР + ИБП + ДГУ** | ИБП держит нагрузку до запуска генератора |

### 10.6. Короткий инженерный вывод

Для оптимизированного варианта серверной на компанию до 200 сотрудников можно принять следующие расчетные параметры:

- **рабочая ИТ-нагрузка:** около **5 кВт**;
- **проектная ИТ-нагрузка с запасом:** около **8 кВт**;
- **охлаждение:** **2 x 8 кВт**, схема **N+1**;
- **ИБП:** **2 x 10 кВА**;
- **автономия ИБП:** **10-15 минут**;
- **ДГУ:** **25-30 кВт**.

Такой набор выглядит технически обоснованным, не избыточным для учебного проекта и при этом оставляет запас на рост компании.

## 11. Чем этот вариант лучше для учебного проекта

Этот вариант лучше выглядит на защите, потому что он:

- не теряет архитектурную правильность;
- не превращает проект в ЦОД “на вырост для корпорации на тысячи человек”;
- остается устойчивым к авариям;
- не использует санкционно-рискованный стек;
- лучше соответствует масштабу компании до 200 сотрудников;
- отвечает на замечание, что тяжелый ERP-контур для такого учебного кейса может быть избыточным.

## 12. Короткий вывод

Для компании до 200 сотрудников с запасом на рост целесообразно использовать **оптимизированную двухплощадочную архитектуру**, где:

- HQ и DR Site сохраняются;
- резервное копирование и offsite/offline-контур сохраняются;
- филиалы и лаборатория остаются защищенными;
- серверный контур уменьшается до 2-3 узлов виртуализации и более компактной СХД;
- вместо тяжелого ERP-подхода применяется **специализированная стоматологическая МИС**.

Именно такой вариант обычно выглядит наиболее убедительно для учебного проекта:  
**не минималистично, но и не чрезмерно дорого для компании такого масштаба.**
